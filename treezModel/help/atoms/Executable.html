Executable

Purpose

The purpose of the atom is to execute a system command, e.g. 
"C:\windows\system32\mshta.exe" javascript:alert(42);close();
The system command typically consists of the path to an executable file and optional arguments (for input, output and logging).

Class name

org.treez.model.atom.executable.Executable;

Work flow

You can start the atom with
  • the play button in the upper right corner of the property page,
  • with the play button in the context menu or
  • remotely with another atom (e.g. in a ).
The will be executed in three steps:
  1. If it has children, those children are executed first. An input file that has been generated by an child can be referenced by the system command in the next step.
  2. The system command is executed. If the executable writes text to the system output, this text will be logged to the Eclipse console.
  3. If the atom has children, those children are executed. This allows to import data that has been generated by the execution of the system command.

Construction

A new is created by:
  • using the context menu of a atom or
  • by calling the corresponding factory method of the atom: 
    Executable executable = models.createExecutable("executable");

Arguments

Executable

  • Executable: The path to the executable file. This path will be automatically put in quotation marks to ensure that paths that include spaces work correctly. Do not include additional arguments here but use the other input fields. The content of the other input fields will be appended to the executable path (separated by spaces). You can have a look at the status section to see a preview of the complete system command that results from the content of all input fields.

Input

  • Input arguments: First part for specifying input arguments. This is typically a key word that tells the executable that the specification of an input file will follow, e.g. "-i" or "-open". Leave this empty if your executable does not require such an input key word. If you use special characters please check if you need to include escape characters, e.g. double slashes or quotation marks instead of single ones. Please consider this also for the other arguments.
  • Input file or folder: Second part for specifying input arguments. This is typically the path to an input file or folder, e.g. "C:/input.txt". If the input file does not yet exist, you can manually create it or use a atom to automatically generate the input file from a before the system command is executed. This path will not automatically be put in quotation marks because some programs do not support quotation marks around the input file path. If you use an input path that contains spaces you might need to wrap it in quotation marks. Leave this empty if your executable does not require an input file.

Output

  • Output arguments: First part for specifying output arguments. This is typically a key word that tells the executable that the specification of an output file will follow, e.g. "-output" or "-o". Leave this empty if your executable does not require such an output key word.
  • Output file or folder: Second part for specifying output arguments. This is typically the path to an output file or folder, e.g. "C:/output.txt". This path will not automatically be put in quotation marks because some programs do not support quotation marks around the output file path. If you use an output path that contains spaces you might need to wrap it in quotation marks. Leave this empty if your executable does not require an output file/folder.

Output modification

The check boxes in the output modification section allow to automatically modify the output file/folder that is specified above. This makes sense if you execute the atom many times, for example in a study, and you want to have a different output location for each execution.

The check boxes allow you to include the execution date and or a study index (=increasing number). You can append them to the name of the (parent) folder, the name of an extra sub folder or to the file name by enabling the corresponding check boxes. If you use the output modification please make sure that your executable is able to automatically create the files/folders if they do not exist.

You can manually reset the study index with the orange button in the upper right corner of the property page. If you remotely run the atom as part of a study, the study index will be reset at the start and then automatically increased while executing the study.

If you are not sure how the check boxes affect the system command, have a look the status section where a preview for the resulting system command is shown. The status section also shows the study index that will be used for the next manual execution. (The actual execution date will be different to the date that is shown in the preview and the study index might be overridden when remotely running the the atom as part of a study.)

Logging

  • Log arguments: First part for specifying logging arguments. This is typically a key word that tells the executable that the specification of a logging file will follow, e.g. "-log" or "-l". Leave this empty if your executable does not require such a logging key word.
  • Log file: Second part for specifying logging arguments. This is typically the path to a logging file, e.g. "C:/log.txt". This path will not automatically be put in quotation marks because some programs do not support quotation marks around the logging file path. If you use a logging path that contains spaces you might need to wrap it in quotation marks.Leave this empty if your executable does not require a log file.
